## 1. 지연된 활성화 핵심 프레임워크 구축

- [x] 1.1 지연된 장치 활성화 상태를 추적하기 위해 프로파일(Profile)/장치 모델 및 뷰모델에 `IsPending` 상태 추가.
- [x] 1.2 `AudioDeviceManager` 등 핵심 클래스에 `IMMNotificationClient`(또는 기존 장치 상태 리스너)를 구현하여 장치 상태 변경 이벤트(예: `OnDeviceStateChanged`) 연결.
- [x] 1.3 장치 상태 변경 이벤트가 메인 UI 및 전환 로직 스레드로 올바르게 전달되는지 확인.

## 2. 장치 전환 로직 업데이트

- [x] 2.1 프로파일 전환 로직을 업데이트하여 실행 전 대상 장치의 사용 가능 여부 확인 기능 추가.
- [x] 2.2 대상 장치가 연결 해제/비활성 상태인 경우 에러나 스킵 처리 대신 프로파일의 상태를 `Pending`으로 설정하고 대기 중인 요청으로 저장.
- [x] 2.3 대상 장치의 상태가 활성화(Active)로 변경될 때 대기 중인 프로파일 전환을 자동으로 실행하는 핸들러 추가.
- [x] 2.4 새로운 프로파일 전환이 요청되면 기존에 대기 중이던 전환 요청을 취소하는 로직 구현.
- [x] 2.5 흩어져 있던 알림(Notification) 발생 코드를 제거하고, 핵심 전환 로직 내 실제 장치 전환 성공 시점에만 공통으로 트리거되도록 이동.

## 3. 메인 UI 프로파일 카드 및 트레이 툴팁 업데이트

- [x] 3.1 `IsPending` 상태를 읽을 수 있도록 프로파일 카드의 데이터 템플릿(DataTemplate/ViewModel) 업데이트.
- [x] 3.2 `IsPending`이 참(true)일 때 프로파일 아이콘 우측 하단의 기존 초록색 체크 표시(Checkmark)를 대체할 로딩 스피너 애니메이션 요소 추가.
- [x] 3.3 로딩 스피너가 대기 상태를 정확히 반영하고 전환이 완료되거나 취소되면 즉시 체크 표시 또는 빈 상태로 돌아가는지 확인.
- [x] 3.4 장치가 대기(Pending) 상태로 진입 시 트레이 아이콘의 이미지는 유지하되, 마우스 오버 툴팁(Tooltip) 텍스트에 대기 중인 프로파일 이름과 상태를 명시하도록 로직 추가.

## 4. 테스트 및 검증

- [x] 4.4 대기 기간 동안 불필요한 CPU 리소스 소모(폴링 루프 등)가 없는지 성능 프로파일러로 검증.
- [x] 4.5 프로파일 전환 실패 시 또는 대기 진입 시 허위 알림이 뜨지 않고, 전환 성공 시점에만 알림이 뜨는지 검증.
