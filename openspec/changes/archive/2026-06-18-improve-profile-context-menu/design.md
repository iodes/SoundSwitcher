## Context

현재 애플리케이션의 메인 UI에서 프로파일 카드를 우클릭하면 컨텍스트 메뉴가 나타납니다. 기존 컨텍스트 메뉴는 기능들이 단순 나열되어 있어 가독성이 떨어지며, 프로파일에 설정된 오디오 장치의 세부 속성을 확인하거나 변경하려면 사용자가 직접 윈도우 소리 설정을 찾아가야 하는 불편함이 있습니다.

## Goals / Non-Goals

**Goals:**
- 프로파일 컨텍스트 메뉴 항목들을 기능별(기본 설정, 아이콘, 속성, 삭제)로 그룹화하여 사용성을 개선.
- 컨텍스트 메뉴에서 해당 프로파일에 연결된 출력/입력 장치의 윈도우 네이티브 속성 창을 직접 열 수 있도록 기능 추가.

**Non-Goals:**
- 애플리케이션 내부에 별도의 자체 장치 속성 UI를 구현하는 것 (유지보수 및 윈도우 자체 기능 활용을 위해 네이티브 창을 띄우는 것으로 한정).
- 시스템 트레이 아이콘의 컨텍스트 메뉴 변경.

## Decisions

**결정 1: 윈도우 네이티브 오디오 장치 속성 창 호출 방식**
- **대안 고려:** C#에서 윈도우 오디오 장치 속성 창을 여는 방법은 여러 가지가 있습니다.
  1. `mmsys.cpl`을 커맨드라인 매개변수와 함께 실행하여 특정 장치 탭 선택.
  2. `Process.Start`로 특정 CLSID나 셸 커맨드 실행.
  3. COM 인터페이스(예: `IPolicyConfig`, `IMMDevice`)를 사용하여 직접 프로퍼티 페이지(Property Sheet) 호출.
- **선택된 방안:** `mmsys.cpl` 실행과 같이 단순히 소리 제어판을 띄우는 것이 아니라, 특정 장치 인스턴스 ID에 대한 "장치 속성" 다이얼로그를 정확히 띄우기 위해 COM 기반 혹은 셸 커맨드를 적절히 활용합니다. 만약 복잡성이 크다면 `mmsys.cpl,,0` 방식보다 더 세밀하게 디바이스 ID를 넘길 수 있는 방법을 탐색합니다. (일반적으로 `rundll32.exe shell32.dll,Control_RunDLL mmsys.cpl,,0` 방식은 특정 디바이스의 속성창만 띄우기엔 한계가 있으므로, 필요시 `IShellFolder`나 P/Invoke를 통한 디바이스 속성 창 다이렉트 호출 고려).

**결정 2: UI 컨텍스트 메뉴 구성 변경**
- WPF/XAML의 `ContextMenu`와 `Separator`를 사용하여 요구사항에 맞게 메뉴를 분리합니다.
- 각 메뉴 아이템에 바인딩할 `ICommand`를 `ProfileViewModel`에 추가하여, 출력 장치/입력 장치가 `null`이 아닐 때만 "속성" 메뉴가 활성화되도록 구현합니다.

## Risks / Trade-offs

- **[Risk]** 특정 오디오 장치의 속성 창을 직접 띄우는 Windows API가 버전에 따라 동작이 다를 수 있음.
  - **Mitigation:** API 호출이 실패할 경우, 사용자가 수동으로 찾을 수 있도록 기본 소리 제어판(`mmsys.cpl`)을 여는 폴백(Fallback) 로직을 구현합니다.
- **[Risk]** 입력 장치가 설정되지 않은 프로파일의 경우 "입력 장치 속성" 메뉴 클릭 시 예외 발생.
  - **Mitigation:** 장치가 없을 경우 메뉴 항목을 비활성화(Disable)하거나 클릭 시 안내 메시지를 표시하도록 `CanExecute` 조건을 설정합니다.
