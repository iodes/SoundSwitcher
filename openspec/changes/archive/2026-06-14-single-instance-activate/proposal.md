## 왜 필요한가 (Why)

현재 SoundSwitcher는 여러 인스턴스가 동시에 실행될 수 있어 리소스 충돌 문제가 발생할 수 있습니다. 또한, 프로그램이 시스템 트레이에 최소화되어 있을 때 새롭게 앱을 실행하거나 `/Activate` 인자를 통해 실행 중인 인스턴스의 창을 활성화할 방법이 없습니다. 이번 변경을 통해 Mutex를 이용한 단일 인스턴스 제한을 적용하고, 추가로 **단순한 IPC(프로세스 간 통신)** 를 도입하여 두 번째 인스턴스가 실행될 때 기존에 실행 중인 인스턴스의 윈도우를 활성화하도록 개선하고자 합니다.

## 무엇이 변경되는가 (What Changes)

- 앱 시작 시 Mutex를 확인하여 SoundSwitcher의 다중 실행을 방지합니다.
- EventWaitHandle을 이용한 간단한 IPC 메커니즘을 추가합니다.
- 첫 번째 인스턴스는 백그라운드에서 활성화 신호(IPC)를 대기합니다.
- Mutex로 인해 기존 인스턴스가 실행 중임을 감지하면, 새로운 인스턴스는 기존 인스턴스에 활성화 신호를 보낸 뒤 즉시 종료됩니다. 신호를 받은 기존 인스턴스는 화면에 창을 표시하고 활성화합니다.

## 기능 (Capabilities)

### 새로운 기능 (New Capabilities)
- `single-instance-activation`: 애플리케이션의 단일 인스턴스 실행을 보장하고, IPC를 통해 중복 실행 시도 시 기존 인스턴스의 윈도우를 활성화하는 메커니즘을 제공합니다.

### 수정된 기능 (Modified Capabilities)

## 영향 (Impact)

- `App.xaml.cs` 등 애플리케이션 시작을 담당하는 코드가 Mutex 및 EventWaitHandle(IPC) 처리를 수행하도록 변경됩니다.
- 프로그램이 중복 실행되었을 때 사용자에게 자연스럽게 기존 창을 보여주는 UX 개선 효과가 있습니다.
