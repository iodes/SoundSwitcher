## 배경 (Context)

SoundSwitcher 애플리케이션은 시스템 트레이(TaskbarIcon)의 우클릭 메뉴로 WPF `ContextMenu`를 사용하고 있습니다. 기본적으로 WPF나 Windows 운영체제는 컨텍스트 메뉴에 슬라이드나 플라이아웃 애니메이션을 적용하는데, 사용자가 단순히 메뉴 항목을 빠르게 선택하고자 할 때 이러한 애니메이션은 UI가 느리게 반응하는 것처럼 느끼게 만들 수 있습니다.

## 목표 및 비목표 (Goals / Non-Goals)

**목표 (Goals):**
- 시스템 트레이 컨텍스트 메뉴가 더 빠르고 반응성 있게 열리도록 개선합니다.
- 트레이 컨텍스트 메뉴에서 슬라이드/플라이아웃 애니메이션을 제거합니다.
- 필요에 따라 단순한 "페이드(Fade)" 애니메이션을 적용하거나, 애니메이션을 완전히 제거("None")합니다.

**비목표 (Non-Goals):**
- 애플리케이션 내 다른 UI 요소들의 애니메이션을 수정하는 것은 포함되지 않습니다.
- 트레이 메뉴 내부의 항목(내용)이나 전체적인 구조를 변경하는 것은 포함되지 않습니다.

## 결정 사항 (Decisions)

**결정 1: WPF 스타일링을 활용한 Popup 애니메이션 수정**
`App.xaml.cs`에 정의된 `_trayContextMenu` 인스턴스에 WPF `Style`을 적용하거나 `ControlTemplate`을 수정하여, `System.Windows.Controls.Primitives.Popup.PopupAnimation` 첨부 속성(Attached Property)을 `PopupAnimation.Fade` 또는 `PopupAnimation.None`으로 설정할 것입니다.

*이유:* WPF의 `ContextMenu`는 내부적으로 `Popup` 컨트롤을 사용합니다. `ContextMenu`를 대상으로 하는 스타일을 정의하고 `Popup.PopupAnimation` 속성을 덮어씌움으로써, 복잡한 Win32 API 호출이나 커스텀 컨트롤 제작 없이도 기본 애니메이션 동작을 손쉽게 변경할 수 있습니다.

*고려했던 대안들:*
- *Win32 API 후킹:* 메뉴 표시 메시지를 가로채어 애니메이션을 억제하는 방식. 이는 단순한 시각적 효과 변경을 위해 도입하기에는 지나치게 복잡합니다.
- *커스텀 팝업 제작:* `ContextMenu` 대신 완전히 새로운 커스텀 팝업 창을 만드는 방식. 키보드 탐색이나 외부 클릭 시 닫힘(light dismiss)과 같은 기본 메뉴 동작을 모두 다시 구현해야 하므로 비효율적입니다.

## 위험 요소 및 고려사항 (Risks / Trade-offs)

- **[위험]** OS 수준의 애니메이션 설정이 WPF의 설정을 강제로 덮어씌울 경우, 스타일 변경이 제대로 적용되지 않을 수 있습니다.
  **완화 방안:** Windows 10 및 11 등 다양한 환경에서 테스트를 거쳐 `PopupAnimation` 설정이 WPF 렌더링 엔진에서 정상적으로 존중되는지 확인합니다.
- **[위험]** 애니메이션을 `None`으로 설정할 경우 메뉴가 나타나는 과정이 너무 갑작스럽고 뻣뻣하게 느껴질 수 있습니다.
  **완화 방안:** 먼저 `Fade` 애니메이션으로 테스트를 진행하고, 여전히 느리다고 판단될 경우에만 `None`으로 변경합니다.
