## Context
현재 SoundSwitcher는 프로파일 전환 요청 시 오디오 장치가 사용 가능한 상태여야 합니다. 사용자가 비활성 상태이거나 연결이 끊긴 장치(예: 전원이 꺼진 블루투스 헤드셋)와 연결된 프로파일로 전환을 시도하면 즉시 실패하거나 건너뛰게 됩니다. 사용자는 먼저 프로파일을 선택(예약)한 뒤 기기의 전원을 켜면 시스템이 장치 활성화 시점에 자동으로 전환해주기를 원합니다. 메인 화면의 프로파일 카드 아이콘은 기존의 초록색 활성 체크 표식 대신 로딩 스피너를 표시하여 이러한 대기 상태를 시각적으로 반영해야 합니다.

## Goals / Non-Goals

**Goals:**
- 사용자가 현재 사용할 수 없는 장치로의 전환을 시도할 수 있도록 허용하고 "대기(Deferred/Pending)" 상태로 진입하도록 합니다.
- UI(특히 메인 화면의 프로파일 카드의 아이콘 영역 및 시스템 트레이 아이콘의 툴팁)에서 대기 상태를 시각적으로 표시합니다.
- 폴링(Polling) 방식 없이 Windows Audio API를 사용하여 CPU 효율적으로 장치의 가용성을 모니터링합니다.
- 대기 중인 장치가 활성화되면 자동으로 전환을 완료합니다.

**Non-Goals:**
- 시스템에 한 번도 연결된 적이 없는 알 수 없는 장치에 대한 대기 전환 지원 (장치 ID가 프로파일에 등록되어 있어야 함).
- 장치가 연결되었다가 즉시 끊기는 등 상태가 불안정한 경우 무한 재시도 처리.

## Decisions

**Decision 1: 장치 상태 모니터링 메커니즘**
- **Choice:** Windows Core Audio API의 `IMMNotificationClient`를 사용하여 장치 상태 변경을 수신합니다.
- **Rationale:** `IMMNotificationClient`는 오디오 장치의 상태 변경(예: `DEVICE_STATE_UNPLUGGED`에서 `DEVICE_STATE_ACTIVE`로 변경) 시 푸시 기반 알림을 제공합니다. 이는 폴링 루프에 의존하지 않으므로 CPU 효율적이며 OS 수준의 이벤트에 반응합니다.
- **Alternatives Considered:** 백그라운드 스레드에서 몇 초마다 `IMMDeviceEnumerator::GetDevice`를 폴링하는 방식. 불필요한 CPU 오버헤드 발생 우려로 거절되었습니다.

**Decision 2: UI 상태 표현**
- **Choice:** ViewModel/Profile 모델에 새로운 `Pending(대기)` 상태를 도입하여 프로파일 카드 UI에서 초록색 체크 마크 대신 로딩 스피너를 렌더링하고, 트레이 아이콘의 툴팁 텍스트를 업데이트하도록 합니다.
- **Rationale:** 사용자는 자신이 클릭한 프로파일 카드가 작업을 등록했으나 대기 중임을 직관적으로 알아야 합니다. 카드의 아이콘 우측 하단 체크 표시를 스피너로 교체하는 것은 이를 위한 효과적이고 기존 디자인과 일관된 UI 패턴입니다. 또한 바탕화면에서도 트레이 아이콘에 마우스를 올려 대기 상태를 확인할 수 있도록 툴팁을 활용합니다.

**Decision 3: 여러 대기 요청 처리**
- **Choice:** 가장 최근의 프로파일 전환 요청만 대기열에 유지됩니다. 사용자가 프로파일 A(대기)를 선택한 뒤 프로파일 B를 선택하면, 시스템은 A에 대한 대기를 중단하고 즉시 B를 시도합니다(B가 비활성화 상태라면 B를 대기).
- **Rationale:** 여러 장치가 순차적으로 켜질 때 발생할 수 있는 예기치 않은 전환을 방지하고, 시스템 상태가 항상 사용자의 가장 최근 의도와 일치하도록 보장합니다.

**Decision 4: 전환 알림(Notification) 트리거 시점 및 로직 공통화**
- **Choice:** 프로파일 전환 알림을 UI의 버튼 클릭/단축키 이벤트 핸들러에서 분리하여, 코어 전환 로직(`AudioDeviceManager` 등)이 장치를 성공적으로 변경한 직후에만 트리거되도록 공통화합니다.
- **Rationale:** 장치 상태를 검증하기 전에 단순 클릭만으로 알림이 발생하면, 전환에 실패하거나 대기 상태에 진입했을 때 허위 피드백이 발생합니다. 실제 전환 성공 이벤트에 알림을 바인딩함으로써 이 문제를 해결하고 로직의 응집도를 높입니다.

## Risks / Trade-offs

- [Risk] COM 스레드 마샬링 문제로 인한 장치 상태 변경 이벤트 누락. → UI를 업데이트하거나 핵심 로직을 트리거해야 하는 경우 `IMMNotificationClient` 콜백이 올바른 스레드(Dispatcher)로 전달되도록 보장하여 완화.
- [Risk] 장치가 빠르게 연결과 해제를 반복(Flapping)하여 다중 전환 시도를 유발. → 처음 활성 상태로 전환될 때 한 번만 전환을 시도하고 전환 시도 후 대기 상태를 해제하여 완화.
